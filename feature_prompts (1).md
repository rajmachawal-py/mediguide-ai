## 📄 Feature 1: PDF Health Card

### What It Does
After triage completes (is_final=true), user taps "Download Health Card" → generates a beautiful PDF on the phone with the full triage summary that can be shown to a doctor at the hospital.

### Use Case
Patient completes triage → reaches hospital → shows PDF to doctor → doctor reads structured assessment in 10 seconds instead of spending 10 minutes asking questions.

### Implementation Prompt

```
FEATURE: PDF Health Card Generator

PROJECT CONTEXT:
- MediGuide AI is a React + FastAPI healthcare app
- Frontend: React 18 + Vite + TailwindCSS at frontend/
- Backend: Python FastAPI at backend/
- After triage, `useChat` hook has: messages[], urgency, urgencyData, summary, language, isFinal
- ChatPage.jsx already has a "Generate Doctor Summary" button (line 160-173) that calls getSummary()
- The summary is returned as JSON from Gemini via POST /api/summary

TASK: Build a PDF Health Card generator

FRONTEND CHANGES:

1. Install jsPDF:
   cd frontend && npm install jspdf

2. Create: frontend/src/services/healthCardGenerator.js
   - Export function: generateHealthCard({ patientName, age, gender, language, urgency, urgencyData, messages, summary })
   - Uses jsPDF to create a styled A5-size PDF containing:
     - Header: "🏥 MediGuide Health Card" with gradient bar
     - Patient Info: Name, Age, Gender, Date/Time
     - Urgency Badge: Color-coded (green/yellow/red) with text
     - Symptoms: Extract user messages from chat history
     - AI Assessment: From summary object
     - Suggested Specialty: From urgencyData.recommend_specialty
     - Disclaimer: "⚠️ This is AI-assisted triage, not a medical diagnosis. Consult a doctor."
     - Footer: "Generated by MediGuide AI • [timestamp]"
   - Support Hindi/Marathi text (jsPDF needs a Unicode font - use Noto Sans Devanagari)
   - Return: triggers browser download of the PDF

3. Modify: frontend/src/pages/ChatPage.jsx
   - After summary is generated (line 176), add a "📄 Download Health Card" button
   - On click: call generateHealthCard() with data from useChat hook
   - Get patient name/age/gender from localStorage or profile API
   - Button style: green gradient, prominent, with download icon

4. Alternative approach (simpler): 
   - Instead of PDF, generate a styled HTML card and use html2canvas to convert to image
   - User can then share via WhatsApp/save to gallery
   - This might be easier for Hindi/Marathi text rendering

DESIGN:
- The Health Card should look professional and medical
- Use the app's dark theme colors but with a WHITE background (for printing)
- Urgency badge should be prominent and color-coded
- Include a border/frame that looks official

DO NOT modify any backend files for this feature - it's purely frontend.
```

## 📸 Feature 2: Visual Symptom Scanner

### What It Does
User uploads a photo (skin rash, wound, swelling) during the chat → image is sent to Gemini 2.5 Flash Vision → AI analyzes the image alongside the conversation → provides visual-aware triage.

### Use Case
Mother sees red patches on her child's arm → can't describe it in words → takes a photo → AI identifies it as ringworm → suggests dermatologist.

### Implementation Prompt

```
FEATURE: Visual Symptom Scanner (Gemini Vision)

PROJECT CONTEXT:
- Backend uses Vertex AI SDK: backend/app/services/gemini_service.py
- Model: gemini-2.5-flash (already supports multimodal image+text input)
- Vertex AI is initialized via _ensure_vertex_init() with service account
- Frontend chat: ChatPage.jsx → useChat hook → POST /api/triage
- Current triage request model (backend/app/models/chat_models.py):
  class TriageRequest:
      symptom: str
      language: str
      history: list[ChatMessage]
- VoiceButton.jsx is in the chat input area at frontend/src/components/chat/

TASK: Add image upload + Gemini Vision analysis to the triage flow

BACKEND CHANGES:

1. Modify: backend/app/models/chat_models.py
   - Add optional field to TriageRequest:
     image_base64: Optional[str] = None  # base64 encoded image data
   - Add optional field to ChatMessage:
     image_base64: Optional[str] = None

2. Modify: backend/app/services/gemini_service.py
   - Update ask_triage() to accept optional image_base64 parameter
   - When image is provided, create a multimodal message:
     from vertexai.generative_models import Part, Image
     - Convert base64 to bytes
     - Create image Part: Part.from_data(data=image_bytes, mime_type="image/jpeg")
     - Send both text Part + image Part in the user message
   - The text part should include:
     "[User has uploaded an image of their symptom. Analyze the image and incorporate 
      your visual assessment into the triage. Describe what you see in the image in 
      the user's language. If it's a skin condition, describe the appearance and 
      suggest possible conditions. Always add disclaimer that visual AI analysis 
      is not a diagnosis.]"

3. Modify: backend/app/routers/triage.py
   - Pass image_base64 from request to ask_triage() if present

FRONTEND CHANGES:

4. Create: frontend/src/components/chat/ImageUploadButton.jsx
   - Camera icon button (same style as VoiceButton)
   - On click: opens file picker (accept="image/*" capture="environment")
   - On mobile: opens camera directly
   - Compresses image to max 1MB before uploading (use canvas resize)
   - Converts to base64
   - Shows image thumbnail preview before sending
   - Calls onImageCapture(base64String) callback

5. Modify: frontend/src/hooks/useChat.js
   - Update sendMessage to accept optional imageBase64 parameter
   - Pass image_base64 in the POST /api/triage request body
   - Store image in message history for display

6. Modify: frontend/src/pages/ChatPage.jsx
   - Add ImageUploadButton next to VoiceButton in the input area (line 192-205)
   - When image is captured, send it with a default text like "Please analyze this image"

7. Modify: frontend/src/components/chat/ChatBubble.jsx
   - If message has an image, render the image thumbnail above the text
   - Style: rounded corners, max-width 200px

IMPORTANT:
- Add disclaimer in Gemini's system prompt: "When analyzing medical images, always 
  state that this is an AI-assisted observation, not a clinical diagnosis."
- Compress images client-side before sending (mobile photos can be 5MB+)
- Use mime_type detection from the file input
```

## 🗣️ Feature 3: Voice Auto-Mode

### What It Does
Toggle switch activates fully hands-free mode: AI speaks → auto-records user's voice → sends to triage → AI speaks response → loop. Zero screen interaction needed.

### Use Case
Elderly illiterate user toggles "Voice Mode" → speaks symptoms → AI responds via speaker → user answers follow-ups by voice → triage completes → urgency spoken aloud. Never touches screen after the first tap.

### Implementation Prompt

```
FEATURE: Voice Auto-Mode (Fully Hands-Free Triage)

PROJECT CONTEXT:
- Voice components exist:
  - frontend/src/components/chat/VoiceButton.jsx — manual push-to-talk STT
  - frontend/src/services/sarvam.js — Sarvam AI STT/TTS API calls
  - frontend/src/services/api.js — has speechToText() and textToSpeech() functions
- Current flow: user taps mic → speaks → release → transcript sent to triage
- TTS exists but is NOT auto-triggered after AI responds
- ChatPage.jsx manages the full chat UI

TASK: Add a "Voice Auto-Mode" toggle that creates a fully hands-free loop

FRONTEND CHANGES:

1. Create: frontend/src/hooks/useVoiceAutoMode.js
   - Custom hook that manages the auto-mode loop:
   
   State:
     - isAutoMode: boolean (toggle on/off)
     - isListening: boolean (currently recording)
     - isSpeaking: boolean (TTS playing)
   
   Flow when enabled:
     a. Play welcome TTS: "नमस्ते, मैं मेडिगाइड हूं। आपको क्या तकलीफ है?"
     b. After TTS finishes → auto-start recording (3-8 seconds of silence detection)
     c. When user stops speaking → send transcript to triage via sendMessage()
     d. When AI responds → auto-play response via TTS (Sarvam bulbul:v3)
     e. After TTS finishes → auto-start recording again (go to step b)
     f. When triage is final (isFinal=true) → speak urgency result → stop auto-mode
   
   Silence detection:
     - Use Web Audio API's AnalyserNode to detect when user stops speaking
     - After 2 seconds of silence → stop recording and submit
     - Maximum recording time: 15 seconds (auto-stop)
   
   Cancel:
     - User can tap the screen anywhere or press a "Stop" button to exit auto-mode
     - This stops any TTS playback and recording

2. Create: frontend/src/components/chat/VoiceAutoModeOverlay.jsx
   - Full-screen overlay that appears when auto-mode is on
   - Shows:
     - Large pulsing mic animation when listening (green pulse)
     - Speaker wave animation when AI is speaking (blue pulse)
     - "Processing..." spinner when waiting for triage response
     - Current transcript text (live, as user speaks)
     - "Tap anywhere to stop" text at bottom
   - Dark overlay with glassmorphism
   - Large "STOP" button at bottom

3. Modify: frontend/src/pages/ChatPage.jsx
   - Add toggle button in header area: "🗣️ Voice Mode" (small toggle/switch)
   - When toggled on: render VoiceAutoModeOverlay on top of chat
   - Pass sendMessage, language, messages, isFinal, urgency to the overlay
   - Messages still appear in chat underneath (for reference)

4. Modify: frontend/src/services/sarvam.js (if needed)
   - Ensure textToSpeech returns an audio Blob that can be played
   - The function should already work — just verify it returns playable audio

TTS INTEGRATION:
- Use Sarvam bulbul:v3 for TTS (already configured in sarvam.js)
- Language mapping: hi → hi-IN, mr → mr-IN, en → en-IN
- Play audio using: new Audio(URL.createObjectURL(audioBlob)).play()

STT INTEGRATION:
- Use existing Sarvam saaras:v3 for STT
- Send recorded audio blob to speechToText() function
- The function is already in frontend/src/services/api.js

SILENCE DETECTION:
- Use MediaRecorder + Web Audio API AnalyserNode
- Check audio levels every 100ms
- If below threshold for 2 seconds → stop recording
- This prevents the app from recording forever

DO NOT modify any backend files for this feature — it's purely frontend.
The existing /api/triage and /api/voice endpoints handle everything.
```
